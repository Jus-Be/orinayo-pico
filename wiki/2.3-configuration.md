This page documents the three primary configuration files that control the behavior of the Orinayo system: `sdkconfig.h` (Bluepad32 settings), `tusb_config.h` (USB MIDI settings), and `btstack_config.h` (Bluetooth stack settings). These files define compile-time constants that control resource allocation, feature enablement, and system limits.

For hardware setup and flashing procedures, see [Building and Flashing](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/Building and Flashing)

 For runtime operational modes that change system behavior without recompilation, see [Operational Modes](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/Operational Modes)

## Bluepad32 Configuration (sdkconfig.h)

The `sdkconfig.h` file configures the Bluepad32 controller library, which handles gamepad HID device discovery and input processing. All macros use the `CONFIG_BLUEPAD32_` prefix.

### Platform and Target Settings

| Macro | Value | Purpose |
| --- | --- | --- |
| `CONFIG_BLUEPAD32_PLATFORM_CUSTOM` | defined | Indicates custom platform implementation (not ESP32) |
| `CONFIG_TARGET_PICO_W` | defined | Specifies Raspberry Pi Pico W as target hardware |

These settings tell Bluepad32 to use the custom platform adapter located in the `bluepad32/` directory rather than the default ESP-IDF platform.

Sources: [sdkconfig.h L4-L5](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/sdkconfig.h#L4-L5)

### Resource Limits

| Macro | Value | Description |
| --- | --- | --- |
| `CONFIG_BLUEPAD32_MAX_DEVICES` | 1 | Maximum simultaneous Bluetooth devices |
| `CONFIG_BLUEPAD32_MAX_ALLOWLIST` | 1 | Maximum allowlist entries for pairing |

These limits are intentionally set to 1 because Orinayo is designed as a single-controller gateway. This reduces memory usage and simplifies the state machine in `pico_bluetooth.c`.

## TinyUSB Configuration (tusb_config.h)

The `tusb_config.h` file configures the USB device stack for MIDI output. This file follows the standard TinyUSB configuration format.

### Core USB Settings

| Macro | Value | Description |
| --- | --- | --- |
| `CFG_TUSB_MCU` | (compiler flag) | Must be defined by build system to specify MCU type |
| `CFG_TUSB_RHPORT0_MODE` | `OPT_MODE_DEVICE` | USB operates in device mode (not host) |
| `CFG_TUSB_OS` | `OPT_OS_PICO` | Integration with Pico SDK |
| `CFG_TUD_ENDPOINT0_SIZE` | 64 | Control endpoint buffer size in bytes |

The `CFG_TUSB_MCU` is set by CMake based on the target platform. For Pico W, this is typically `OPT_MCU_RP2040`.

Sources: [tusb_config.h L38-L76](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/tusb_config.h#L38-L76)

### MIDI Buffer Sizes

| Macro | Value | Purpose |
| --- | --- | --- |
| `CFG_TUD_MIDI_RX_BUFSIZE` | 512 (high-speed) or 64 (full-speed) | USB to Pico MIDI buffer |
| `CFG_TUD_MIDI_TX_BUFSIZE` | 512 (high-speed) or 64 (full-speed) | Pico to USB MIDI buffer |

The buffer sizes are conditional based on USB speed capabilities of the target MCU. The RP2040 in Pico W operates at full-speed (12 Mbps), so the actual value used is 64 bytes.

Sources: [tusb_config.h L87-L88](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/tusb_config.h#L87-L88)

The CYW43 wireless chip shares a single SPI bus for both WiFi and Bluetooth. To prevent buffer overruns, BTstack is configured with:

1. **Limited outgoing buffers** ([btstack_config.h L62-L63](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/btstack_config.h#L62-L63) ): Only 3 ACL and 3 SCO packets queued at once.
2. **Controller-to-host flow control** ([btstack_config.h L66-L70](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/btstack_config.h#L66-L70) ): The CYW43 can throttle incoming data when the host is busy.

This configuration is critical for maintaining stable Bluetooth connections during high MIDI traffic periods.

Sources: [btstack_config.h L61-L70](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/btstack_config.h#L61-L70)

### Storage and Persistence

| Macro | Value | Purpose |
| --- | --- | --- |
| `NVM_NUM_DEVICE_DB_ENTRIES` | 16 | BLE device records in flash |
| `NVM_NUM_LINK_KEYS` | 16 | Classic Bluetooth pairing keys |
| `MAX_ATT_DB_SIZE` | 512 | Static ATT database for GATT |

These settings control the flash-based persistent storage for bonded devices. The system uses the "TLV on top of Flash Sector interface" mentioned at [btstack_config.h L72-L74](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/btstack_config.h#L72-L74)

## Common Configuration Changes

### Increasing Maximum Devices

To support multiple simultaneous Bluetooth controllers, modify `sdkconfig.h`:

```python
#define CONFIG_BLUEPAD32_MAX_DEVICES 2  // Changed from 1#define CONFIG_BLUEPAD32_MAX_ALLOWLIST 2  // Changed from 1
```

Also adjust BTstack limits in `btstack_config.h`:

* Increase `MAX_NR_HCI_CONNECTIONS` to accommodate more devices.
* Increase `MAX_NR_L2CAP_CHANNELS` proportionally (add 1-2 per device).

Sources: [sdkconfig.h L8-L9](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/sdkconfig.h#L8-L9)

 [btstack_config.h L47-L51](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/btstack_config.h#L47-L51)

### Adjusting Log Verbosity

To reduce console output during operation, change `sdkconfig.h`:

```
#define CONFIG_BLUEPAD32_LOG_LEVEL 2  // ERROR only (was 0 = INFO)
```

This is useful for production deployments where diagnostic output is not needed. Additionally, the `debug.h` file provides `PICO_DEBUG_MODE` which defaults to `1` (enabled) and controls the behavior of `PICO_DEBUG`, `PICO_INFO`, and `PICO_ERROR` macros.

Sources: [sdkconfig.h L6](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/sdkconfig.h#L6-L6)

 [debug.h L7-L22](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/debug.h#L7-L22)

### USB MIDI Buffer Tuning

If experiencing USB MIDI buffer overruns during high-throughput scenarios (e.g., dense drum patterns), increase the buffer at `tusb_config.h`:

```python
#define CFG_TUD_MIDI_TX_BUFSIZE   128  // Increased from 64
```

Note that this consumes additional RAM and may require adjusting other buffers to maintain stability.

Sources: [tusb_config.h L88](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/tusb_config.h#L88-L88)

## Synchronization Primitives

The system utilizes a lightweight single-writer / multiple-reader sequence lock defined in `seqlock.h` to manage shared state between cores (e.g., between the Bluetooth core and the USB/Processing core).

Title: Seqlock Implementation

```mermaid
flowchart TD

R1["v1 = atomic_load_explicit(seq, memory_order_acquire)"]
R2["Check if v1 is odd"]
R3["Copy data to destination"]
R4["v2 = atomic_load_explicit(seq, memory_order_relaxed)"]
R5["Check if v1 == v2"]
W1["atomic_fetch_add_explicit(seq, 1u, memory_order_relaxed)"]
W2["g_ctrl.data = *src"]
W3["atomic_fetch_add_explicit(seq, 1u, memory_order_release)"]

W1 --> W2
W2 --> W3

subgraph subGraph2 ["Writer - seqlock_write_end"]
    W3
end

subgraph subGraph1 ["Payload Update"]
    W2
end

subgraph subGraph0 ["Writer - seqlock_write_begin"]
    W1
end

subgraph subGraph3 ["Reader - SEQLOCK_TRY_READ"]
    R1
    R2
    R3
    R4
    R5
    R1 --> R2
    R2 --> R3
    R3 --> R4
    R4 --> R5
end
```

The `SEQLOCK_DECL` macro is used to declare protected structures, and `SEQLOCK_TRY_READ` allows for consistent snapshots without disabling interrupts.
