# Data Flow Pipeline

This page traces the complete data flow from Bluetooth input through musical processing to dual MIDI output. It documents how raw controller inputs are transformed into MIDI messages through multiple processing stages, including the critical decision points that route data through different processing paths.

For information about specific operational modes (Arranger, MODX, SeqTrak, etc.), see [Operational Modes](./4.4-operational-modes.md). For details on the looper's internal workings, see [Step Sequencer](./5.2-step-sequencer.md). For Bluetooth connection establishment, see [BLE Device Management](./4.1-ble-device-management.md).

## Overview

The system implements a multi-stage pipeline that processes data from Bluetooth controllers and translates it into MIDI output. Data flows through six major stages:

| Stage | Purpose | Key Components |
| --- | --- | --- |
| 1. Connection & Discovery | Establish BLE/Classic BT connection | `uni_bt.h`, `uni_hid_device.h` |
| 2. Input Acquisition | Receive HID reports or GATT notifications | `gamepad_bluetooth_handle_data`, `ble_midi_controller.c` |
| 3. Button Mapping | Parse raw input into button states | `pico_bluetooth.c` state variables |
| 4. Mode Selection | Route to appropriate handler | Mode flags (`enable_*`) |
| 5. Musical Translation | Convert buttons to MIDI concepts | `play_chord`, `chord_chat` lookup table |
| 6. Output Generation | Create and transmit MIDI messages | `midi_n_stream_write` |

An optional seventh stage, **Sequencer Processing**, is inserted when `enable_midi_drums` is true, adding algorithmic enhancement via the looper subsystem.

Sources: [pico_bluetooth.c L1-L200](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L1-L200)

## Stage 2: Input Acquisition

### Dual Input Paths

The system supports two distinct input paths that converge on the same button state variables:

```mermaid
flowchart TD

CtlData["uni_gamepad_t<br>gamepad"]
ExtractA["Extract button bits"]
VarsA["Button state variables"]
GATT["GATT Notification<br>ble_midi_controller.c"]
ParseB["Parse MIDI Packets"]
VarsB["Button state variables"]
Converge["midi_process_state()"]

VarsA --> Converge
VarsB --> Converge

subgraph subGraph1 ["Path B: BLE MIDI Devices"]
    GATT
    ParseB
    VarsB
    GATT --> ParseB
    ParseB --> VarsB
end

subgraph subGraph0 ["Path A: HID Controllers"]
    CtlData
    ExtractA
    VarsA
    CtlData --> ExtractA
    ExtractA --> VarsA
end
```

**Path A** processes standard gamepads, extracting states into variables like `but0` through `but9` and `dpad_*`.

**Path B** processes specialized BLE MIDI controllers (Liberlive, Sonic Master) by parsing MIDI data into equivalent fret and strum states.

### Fret and Strum Button Processing

Fret buttons (green, red, yellow, blue, orange) are used to determine the active chord. The system tracks:

* **Strum patterns**: `active_strum_pattern` [pico_bluetooth.c L122](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L122-L122)  selects arpeggio sequences.
* **Neck positions**: `active_neck_pos` [pico_bluetooth.c L123](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L123-L123)  determines the octave/voicing.

Sources: [pico_bluetooth.c L96-L123](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L96-L123)

## Stage 5: Musical Translation

### Chord Detection System

The `play_chord()` function [pico_bluetooth.c L214](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L214-L214)

 translates fret combinations into MIDI notes. It uses the `chord_chat` lookup table [pico_bluetooth.c L231-L244](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L231-L244)

 to determine specific guitar fingerings (Root, Type, Fret Positions).

#### Chord Matrix Structure

`chord_chat[12][3][6]` represents:

* 12 Root notes (C, C#, D...)
* 3 Types (Major, Minor, 7th)
* 6 Strings (Fret number per string)

Sources: [pico_bluetooth.c L214-L244](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L214-L244)

### Strum Pattern Application

The `strum_pattern[14][12][6]` [pico_bluetooth.c L246](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L246-L246)

 table defines how chords are arpeggiated when a strum action occurs, mapping specific steps in a pattern to chord notes.

Sources: [pico_bluetooth.c L122-L246](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L122-L246)

### Step Processing Pipeline


For each step:

1. `looper_perform_step()` [looper.c L163](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.c#L163-L163)  checks the active track patterns.
2. `ghost_note_modulate_base_velocity()` [looper.c L172](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.c#L172-L172)  calculates dynamic velocity.
3. `note_scheduler_schedule_note()` [looper.c L173](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.c#L173-L173)  queues the event with microsecond precision.

## Stage 8: Dual Output Transmission

### The Output Bottleneck

The `midi_n_stream_write()` function [main.c L165](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/main.c#L165-L165)

 is the central dispatcher for all MIDI data. It simultaneously writes to:

1. **USB MIDI**: via TinyUSB `tud_midi_n_stream_write`.
2. **UART MIDI**: via `uart_write_blocking`.

Sources: [main.c L165](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/main.c#L165-L165)
