## Purpose and Scope

This document describes the Bluetooth Low Energy (BLE) device management subsystem, covering device scanning, discovery, connection establishment, pairing, security, and connection lifecycle management. The system supports two primary classes of devices:

1. **HID over GATT (HoG)**: Generic gamepads and controllers managed via Bluepad32 in [bluepad32/bt/uni_bt_le.c](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/bluepad32/bt/uni_bt_le.c)
2. **BLE MIDI Peripherals**: Standards-compliant MIDI devices (e.g., keyboards, wind controllers) managed by a specialized client library in [pico-w-ble-midi-lib/ble_midi_client.c](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico-w-ble-midi-lib/ble_midi_client.c)  and integrated via [ble_midi_controller.c](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ble_midi_controller.c)

For information about HID-to-MIDI translation after devices are connected, see [HID to MIDI Translation](./4.2-hid-to-midi-translation.md). For specialized MIDI device protocols, see [Specialized MIDI Devices](./4.5-specialized-midi-devices.md).

## Connection Lifecycle Overview

The BLE connection lifecycle follows a specific sequence of states and events, coordinated by multiple packet handlers.

### Connection State Machine

The following diagram illustrates the transition from initial scanning to a "Ready" device state for both HID and MIDI devices.

```mermaid

    Connecting --> Connected : "HCI_SUBEVENT_LE_CONNECTION_COMPLETE"
    Connected --> Pairing : "sm_request_pairing()"
    Connected --> Reencryption : "SM_EVENT_REENCRYPTION_STARTED"
    Pairing --> PairingComplete : "SM_EVENT_PAIRING_COMPLETE"

    PairingComplete --> ServiceDiscovery : "GATT Service Query"
    ReencryptComplete --> ServiceDiscovery : "SM_EVENT_REENCRYPTION_COMPLETE"
    ServiceDiscovery --> Ready : "GATTSERVICE_SUBEVENT_HID_SERVICE_CONNECTED / BLEMC_WAIT_FOR_MIDI_DATA_RX"
    Ready --> [*] : "HCI_EVENT_DISCONNECTION_COMPLETE"
```

## Connection Establishment

### Connection Initiation

The system stops scanning before initiating a connection to improve reliability.

```mermaid
sequenceDiagram
  participant ble_midi_controller_poll()
  participant ble_midi_client.c
  participant GAP Layer
  participant hci_event_packet_handler

  ble_midi_controller_poll()->>ble_midi_client.c: "ble_midi_client_request_connect(1)"
  ble_midi_client.c->>GAP Layer: "gap_stop_scan()"
  ble_midi_client.c->>GAP Layer: "gap_connect(addr, type)"
  GAP Layer-->>hci_event_packet_handler: "HCI_SUBEVENT_LE_CONNECTION_COMPLETE"
  hci_event_packet_handler->>ble_midi_client.c: "Update con_handle"
  hci_event_packet_handler->>ble_midi_client.c: "state = BLEMC_WAIT_FOR_SERVICES"
  ble_midi_client.c->>GAP Layer: "gatt_client_discover_primary_services_by_uuid16(MIDI_UUID)"
```

**Scan Resume Logic:**
For HID devices, `resume_scanning_hint()` restarts scanning if it was active before the connection [bluepad32/bt/uni_bt_le.c L184-L190](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/bluepad32/bt/uni_bt_le.c#L184-L190)

 For MIDI, the `ble_midi_controller_poll()` function manages reconnection every `BMC_RECONNECT_INTERVAL_MS` if the state is not `BMC_STATE_READY` [ble_midi_controller.c L56-L72](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ble_midi_controller.c#L56-L72)

## Service Discovery

### MIDI Service Flow

For standard MIDI devices, the discovery process is strictly sequenced via a state machine in `ble_midi_client.c`.

```mermaid
sequenceDiagram

  participant GATT Client

  note over ble_midi_client.c: "State: BLEMC_WAIT_FOR_SERVICES"
  ble_midi_client.c->>GATT Client: "gatt_client_discover_primary_services_by_uuid16(0x03B8...)"
  GATT Client-->>ble_midi_client.c: "GATT_EVENT_SERVICE_QUERY_RESULT"
  note over ble_midi_client.c: "State: BLEMC_WAIT_FOR_CHARACTERISTICS"
  ble_midi_client.c->>GATT Client: "gatt_client_discover_characteristics_for_service_by_uuid128(MIDI_DATA_IO)"
  GATT Client-->>ble_midi_client.c: "GATT_EVENT_CHARACTERISTIC_QUERY_RESULT"

  GATT Client-->>ble_midi_client.c: "GATT_EVENT_QUERY_COMPLETE"
  note over ble_midi_client.c: "State: BLEMC_WAIT_FOR_MIDI_DATA_RX"
```

## Configuration and Setup

### Initialization Sequence

The BLE subsystems are initialized through a series of setup calls before the BTstack power-on.

1. **HID Setup**: `uni_bt_le_setup()` registers the HID SM handler and initializes the HIDS client [bluepad32/bt/uni_bt_le.c L1447-L1496](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/bluepad32/bt/uni_bt_le.c#L1447-L1496)
2. **MIDI Setup**: `ble_midi_controller_init()` calls `ble_midi_client_init()` with the profile name "Orinayo" [ble_midi_controller.c L49-L50](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ble_midi_controller.c#L49-L50)  [ble_midi_controller.c L215](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ble_midi_controller.c#L215-L215)
3. **HCI Power**: `hci_power_control(HCI_POWER_ON)` starts the stack [bluepad32/bt/uni_bt_setup.c L163](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/bluepad32/bt/uni_bt_setup.c#L163-L163)

**Runtime Polling:**
The application must call `ble_midi_controller_poll()` in the main loop [ble_midi_controller.c L237](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ble_midi_controller.c#L237-L237)

 This function drains the MIDI receive buffer using `ble_midi_client_stream_read()` and dispatches events to handlers like `bmc_on_note_on()` [ble_midi_controller.c L270-L300](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ble_midi_controller.c#L270-L300)
