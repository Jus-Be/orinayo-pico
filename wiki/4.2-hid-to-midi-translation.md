This page documents how gamepad controller button states are translated into MIDI commands in the Orinayo system. The translation occurs primarily in [pico_bluetooth.c](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c)

 and handles both standard HID gamepad controllers (via Bluepad32) and specialized BLE MIDI devices (Liberlive, Sonic Master).

For information about chord generation algorithms and the `chord_chat` lookup table, see [Chord Generation System](./4.3-chord-generation-system.md). For details on the five operational modes that affect MIDI output, see [Operational Modes](./4.4-operational-modes.md). For specialized BLE MIDI device handling, see [Specialized MIDI Devices](./4.5-specialized-midi-devices.md).

## Overview

The HID to MIDI translation system acts as an interpreter between physical controller inputs and musical MIDI commands. It maintains button state variables, implements a complex state machine for processing button combinations, and generates appropriate MIDI messages based on the current operational mode.

**Key Components:**

* **Button State Variables**: Global variables tracking current button states.
* **State Machine**: `midi_bluetooth_handle_data()` processes button state changes.
* **Chord Detection**: `play_chord()` translates fret combinations to MIDI notes.
* **Mode Switching**: Configuration system enabling different musical behaviors.

Sources: [pico_bluetooth.c L1-L188](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L1-L188)

## Main State Machine

The `midi_bluetooth_handle_data()` function implements a large sequential state machine that processes button state changes in priority order. It uses early returns to ensure only one action is processed per invocation.

```mermaid
stateDiagram-v2
    [*] --> CheckProcessing : "Button Event"
    CheckProcessing --> Exit : "pitch button changed"
    CheckProcessing --> SetFlag : "finished_processing == true"
    SetFlag --> CheckFrets : "Set finished_processing = false"
    CheckFrets --> FretChange : "pitch button changed"
    CheckFrets --> CheckPitch : "No fret change"
    FretChange --> ProcessFretLogic : "Lines 352-373"
    ProcessFretLogic --> ModeActions : "Fret + pitch combinations"
    ModeActions --> ReturnTrue : "Set active_strum_pattern"
    CheckPitch --> PitchChange : "song_key button changed"
    CheckPitch --> CheckSongKey : "No pitch change"
    PitchChange --> ProcessPitch : "Lines 372-542"
    ProcessPitch --> StrumStyles : "Set active_strum_pattern"
    ProcessPitch --> ReturnTrue : "Set active_strum_pattern"
    CheckSongKey --> KeyChange : "song_key button changed"
    CheckSongKey --> CheckStart : "No key change"
    KeyChange --> SetTranspose : "Lines 546-560"
    SetTranspose --> ReturnTrue : "Lines 546-560"
    CheckStart --> CheckDpad : "Lines 564-596"
    CheckDpad --> CheckLogo : "Lines 601-739"
    CheckLogo --> CheckStarpower : "Lines 741-839"
    CheckStarpower --> CheckMenu : "Lines 841-975"
    CheckMenu --> CheckConfig : "Lines 977-998"
    CheckConfig --> CheckJoystick : "Lines 1000-1092"
    CheckJoystick --> CheckKnobs : "Lines 1101-1219"
    CheckKnobs --> CheckStrumRight : "Lines 1221-1244"
    CheckStrumRight --> StrumRightChange : "dpad_right changed"
    CheckStrumRight --> CheckStrumLeft : "No change"
    StrumRightChange --> PlayChordUp : "stop_chord() then play_chord(true, true)"
    PlayChordUp --> ReturnTrue
    CheckStrumLeft --> StrumLeftChange : "dpad_left changed"
    CheckStrumLeft --> FinalReturn : "No change"
    StrumLeftChange --> PlayChordDown : "stop_chord() then play_chord(true, false)"
    PlayChordDown --> ReturnTrue
    ReturnTrue --> [*] : "Return and wait for next event"
    FinalReturn --> [*]
    Exit --> [*]
```

**Processing Sequence:**

1. **Gate Check** (lines 328-329): Return immediately if previous processing incomplete (`finished_processing` flag).
2. **Fret Buttons** (lines 352-541): Highest priority - updates `green`, `red`, `yellow`, `blue`, `orange`.
3. **Pitch Button** (lines 372-542): Processes strum pattern selection with fret modifiers.
4. **Song Key** (lines 546-560): Direct key transposition via fret + `but7`.
5. **Logo Button** (lines 601-739): Start/stop sequencer or arranger.
6. **Starpower** (lines 741-839): Style section selection (Intro, Fill, etc.).
7. **Menu Button** (lines 841-975): Registration/style group selection.
8. **Config Button** (lines 977-998): Operational mode switching.
9. **Joystick** (lines 1000-1092): Tempo presets or fill-in triggers.
10. **Knobs** (lines 1101-1219): Ghost note intensity, recording mode.
11. **Strum Buttons** (lines 1221-1271): Lowest priority - triggers chord playback via `play_chord()`.

Sources: [pico_bluetooth.c L327-L1274](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L327-L1274)

## Strum Button Processing

Strum buttons (d-pad left/right) trigger chord playback. They are processed last in the state machine to allow fret button changes to take precedence.

  participant midi_bluetooth_handle_data

    midi_bluetooth_handle_data->>stop_chord(): Stop previous chord
    stop_chord()->>MIDI Output: Note Off messages

## BLE MIDI Device Handling

Specialized BLE MIDI devices (Liberlive, Sonic Master) bypass the standard HID gamepad path and are handled directly in `ble_midi_controller.c`. The `pico_bluetooth.c` file does not directly contain the GATT client event handling logic for these devices. Instead, it relies on the `ble_midi_controller.h` [pico_bluetooth.c L2](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/pico_bluetooth.c#L2-L2)

 module.
