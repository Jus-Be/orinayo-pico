This page documents the algorithmic note generation system that augments user-recorded drum patterns with additional "ghost" notes. These generated notes add humanization, rhythmic complexity, and dynamic fills to the step sequencer patterns.

For information about the step sequencer itself, see [Step Sequencer](./5.2-step-sequencer.md). For note timing and scheduling, see [Note Scheduler](./5.5-note-scheduler.md). For clock synchronization, see [Clock Synchronization](./5.3-clock-synchronization.md).

## Purpose and Scope

The ghost note generation system provides three distinct algorithms that automatically add MIDI note events to the 32-step looper patterns:

1. **Euclidean Distribution** - Evenly spaces additional notes across empty steps [ghost_note.c L168-L182](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L168-L182)
2. **Boundary Notes** - Adds grace notes immediately before/after user notes [ghost_note.c L185-L204](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L185-L204)
3. **Fill-in Patterns** - Generates drum fills at phrase endings based on pattern density [ghost_note.c L226-L254](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L226-L254)

All generated notes are probabilistic and controlled by a master `ghost_intensity` parameter that acts as a global volume control for algorithmic enhancement [ghost_note.c L21-L27](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L21-L27)

 The system operates per-track, allowing different drums to have different ghost note characteristics.

The ghost note system operates in two phases: **generation** (when patterns are created or updated) and **evaluation** (every step during playback). Generation algorithms populate the `ghost_notes` and `fill_pattern` arrays with probability values. During playback, these probabilities are evaluated against the master `ghost_intensity` and a random sample to determine if each ghost note fires.

### Per-Step Ghost Note Storage

Each track maintains a `ghost_notes[32]` array where each element contains:

* `probability` (0-100): The likelihood this ghost note should fire [looper.h L73](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.h#L73-L73)
* `rand_sample` (0-99): A pre-generated random value for evaluation [looper.h L74](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.h#L74-L74)

The separation of `probability` and `rand_sample` allows deterministic pattern regeneration while maintaining randomness in note placement.

## Euclidean Algorithm

The Euclidean algorithm distributes ghost notes evenly across the 32-step pattern, filling sparse tracks to a target density. It is based on Bresenham's line algorithm, commonly used for Euclidean rhythms in music.

### Algorithm Flow

```mermaid
flowchart TD

Start["ghost_note_create(track)"]
Count["count_user_notes()<br>Count existing notes"]
CheckEmpty["n == 0 or<br>n >= 32?"]
Exit1["Return"]

ClampTarget["target = clamp(n + extra, 1, k_max)"]
CalcPhase["phase_step = 32 / target<br>phase_offset = rand() % phase_step"]
Apply["apply_euclidean_ghost_notes()"]
Loop["For i = 0 to 31"]
Accumulate["accumulator += target"]
CheckAccum["accumulator >= 32?"]
SubAccum["accumulator -= 32<br>pos = (i + offset) % 32"]
CheckEmpty2["pattern[pos]<br>already set?"]
SetGhost["ghost_notes[pos].probability =<br>euclidean.probability * (1 - density)<br>ghost_notes[pos].rand_sample = rand()"]
Skip["Skip"]
Next["Continue"]
Exit2["Return"]
```

### Implementation Details

The function `add_euclidean_ghost_notes()` [ghost_note.c L168-L182](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L168-L182)

 implements this algorithm:

1. **Count Existing Notes**: Determine how many user notes exist via `count_user_notes()` [ghost_note.c L125-L132](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L125-L132)
2. **Calculate Target Density**: Based on `k_sufficient` threshold.
3. **Compute Extra Notes**: Use the formula in `calculate_extra_note_count()` [ghost_note.c L135-L143](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L135-L143)
4. **Apply Bresenham Accumulator**: Distribute `target` notes evenly across 32 steps in `apply_euclidean_ghost_notes()` [ghost_note.c L146-L165](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L146-L165)
5. **Set Probabilities**: Each candidate position gets a probability adjusted by pattern density.

The algorithm ensures that:

* Sparse patterns (< 6 notes) receive more ghost notes.
* Dense patterns (≥ 6 notes) receive few or no ghost notes.
* Ghost notes avoid steps where user notes already exist.
* Probability decreases as pattern density increases: `probability * (1.0 - density)` [ghost_note.c L158-L159](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L158-L159)

**Sources:** [ghost_note.c L124-L182](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L124-L182)

## Boundary Notes

Boundary notes add grace notes immediately adjacent to user-recorded notes, creating leading and trailing embellishments. This algorithm operates independently from Euclidean distribution.

### Algorithm Logic

```mermaid

Done["Return"]
```

### Implementation

The `add_boundary_notes()` function [ghost_note.c L185-L204](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L185-L204)

 scans each user note and attempts to place ghost notes at positions `i-1` and `i+1` (with wraparound for step 0 and 31):

* **Before Note** (10% probability): Creates an anticipation or pickup [ghost_note.c L192-L195](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L192-L195)
* **After Note** (50% probability): Creates a trailing grace note or flam [ghost_note.c L199-L201](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L199-L201)

The function only sets boundary notes if:

1. A user note exists at position `i`.
2. The adjacent position has no user note.
3. The adjacent position has no existing ghost note (`rand_sample == 0`).

This prevents overwriting Euclidean ghost notes and avoids double-triggering on consecutive user notes.

**Sources:** [ghost_note.c L185-L204](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L185-L204)

## Fill-in Patterns

Fill-in patterns generate drum fills at the end of musical phrases, typically on tracks 0 (kick) and 1 (snare). Unlike Euclidean and boundary notes, fills are stored in a separate `fill_pattern[32]` boolean array.

### Trigger Conditions

```mermaid
stateDiagram-v2
    [*] --> Waiting : "pending_fill_request == true"
    Waiting --> Automatic : "ghost_bar_counter == (interval_bar - 2)AND step 0AND pattern_density() > 0"
    Waiting --> Manual : "pending_fill_request == true"
    Automatic --> Generating : "add_fillin_notes()"
    Manual --> GeneratingNow : "add_fillin_notes_now()"
    Generating --> Complete : "add_fillin_notes()"
    GeneratingNow --> Complete : "add_fillin_notes_now()"
    Complete --> Waiting : "pending_fill_request == true"
```

Fills are triggered in two ways:

1. **Automatic**: Every N bars (default: 4), starting 2 bars before the cycle completes [ghost_note.c L337-L340](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L337-L340)
2. **Manual**: When `ghost_note_set_pending_fill_request()` is called (e.g., user button press during external clock sync) [ghost_note.h L50](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.h#L50-L50)

### Fill Generation Algorithm

The fill generation process [ghost_note.c L226-L254](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L226-L254)

 involves several steps:

1. **Calculate Fill Start Position**: Use a Gaussian distribution centered at step 15 via `rand_normal()` [ghost_note.c L111-L114](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L111-L114) : `fill_start = 32 - abs(rand_normal(start_mean, start_sd))` [ghost_note.c L230](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L230-L230) This creates fills that typically start 3/4 through the pattern.
2. **Compute Density Window**: For each position, calculate local note density in a ±8 step window via `track_window_density()` [ghost_note.c L206-L213](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L206-L213)
3. **Set Fill Probabilities**: For tracks 0 and 1, from `fill_start` to step 31: * If no existing ghost note, set probability based on inverse density: `(1.0 - density) * 0.25` [ghost_note.c L242](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L242-L242) * If ghost note probability exceeds threshold, set `fill_pattern[i] = true` with fill probability [ghost_note.c L247-L249](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L247-L249)
4. **Density-Based Placement**: Fills are placed in sparse regions, avoiding dense areas.

## Probability Evaluation System

Ghost notes are evaluated during playback using a two-stage probability system that compares stored probability values against the master `ghost_intensity` parameter.

### Evaluation Formula

```mermaid

Normalize1["÷ 100"]
Intensity["ghost_intensity<br>(0.843)"]
Multiply["×"]
RandSample["ghost_notes[step].rand_sample<br>(0-99)"]
Normalize2["÷ 100"]
Compare["Unsupported markdown: blockquote"]
Play["Play ghost note"]
Skip["Skip"]

Stored --> Normalize1
Intensity --> Multiply
Normalize1 --> Multiply
RandSample --> Normalize2
Multiply --> Compare
Normalize2 --> Compare
Compare --> Play
Compare --> Skip
```

The evaluation occurs in `looper_perform_step()` [looper.c L162-L194](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.c#L162-L194)

:

```
bool ghost_note_on = ((float)tracks[i].ghost_notes[looper_status.current_step].probability / 100.0f) *                       params->ghost_intensity >                       (float)tracks[i].ghost_notes[looper_status.current_step].rand_sample / 100.0f;
```

This formula allows the master `ghost_intensity` to act as a volume control:

* `ghost_intensity = 1.0`: All ghost notes with probability > rand_sample play.
* `ghost_intensity = 0.5`: Only ghost notes with 2× probability > rand_sample play.
* `ghost_intensity = 0.0`: No ghost notes play.

### Fill Pattern Evaluation

Fill patterns use a simpler boolean evaluation [looper.h L76](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.h#L76-L76)

:

```
if (tracks[i].fill_pattern[looper_status.current_step] && !note_on)    note_scheduler_schedule_note(now + swing_offset_us, tracks[i].channel, tracks[i].note, 0x7f);
```

Fills always play at maximum velocity (0x7f) and only fire if no user note exists at that step.

**Sources:** [looper.c L162-L194](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.c#L162-L194)

## Swing Timing

Ghost notes participate in the swing timing system, which delays odd-numbered steps to create a "shuffle" feel.

### Swing Offset Calculation

The `looper_get_swing_offset_us()` function [looper.c L148-L158](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.c#L148-L158)

 computes timing offsets for odd steps:

1. Calculate the duration of a 2-step pair (8th note): `pair_length = looper_status.step_period_ms * 2.0f` [looper.c L151](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.c#L151-L151)
2. Compute offset as the excess beyond 50%: `offset_ms = pair_length * (swing_ratio - 0.5f)` [looper.c L154](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/looper.c#L154-L154)
3. Convert to microseconds for scheduling.

### Dynamic Swing Modulation

The `ghost_note_modulate_swing_ratio()` function [ghost_note.c L69-L88](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L69-L88)

 dynamically adjusts swing based on `ghost_intensity`:

* `ghost_intensity < 0.5`: No swing (0.5) [ghost_note.c L72-L73](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L72-L73)
* `ghost_intensity >= 0.5`: Progressive swing from 0.5 to 0.65 using `powf(t, 7.0f)` [ghost_note.c L76](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L76-L76)
* Adds subtle LFO modulation: `sinf(phase + M_PI_2) * 0.01f` [ghost_note.c L79](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L79-L79)

## Maintenance and Lifecycle

The ghost note system integrates with the looper through the `ghost_note_maintenance_step()` function [ghost_note.c L319-L348](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L319-L348)

 called every step during playback.

### Maintenance Cycle

1. **Bar Counter**: Increments at step 0 [ghost_note.c L321-L323](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/ghost_note.c#L321-L323)
