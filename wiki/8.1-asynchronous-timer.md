## Purpose and Scope

The Asynchronous Timer subsystem provides a unified abstraction for scheduling time-based work in the Orinayo system. It encapsulates platform-specific asynchronous execution contexts and exposes a single interface for scheduling periodic tasks, delayed callbacks, and precisely-timed operations.

This document covers the `async_timer` module's architecture, conditional compilation strategy, and integration patterns. For information about the musical timing system that consumes this infrastructure, see [Clock Synchronization](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/Clock Synchronization)

 For details on microsecond-precision note scheduling built on top of this system, see [Note Scheduler](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/Note Scheduler)

## Worker Callback Pattern

The async context system processes time-based work through the `async_at_time_worker_t` structure:

**Worker Structure and Lifecycle**

```mermaid
stateDiagram-v2
    [*] --> Unscheduled : "Worker created"
    Unscheduled --> Scheduled : "async_context_remove_at_time_worker()"
    Scheduled --> Executing : "Time reached"
    Executing --> Callback : "do_work() invoked"
    Callback --> Unscheduled : "async_context_remove_at_time_worker()"
    Scheduled --> Unscheduled : "async_context_remove_at_time_worker()"
```

All time-based operations follow this pattern:

1. **Define worker:** Allocate `async_at_time_worker_t` with `.do_work` function pointer.
2. **Schedule:** Call `async_context_add_at_time_worker_at()` or `async_context_add_at_time_worker_in_ms()`.
3. **Execute:** System invokes callback at specified time.
4. **Reschedule (optional):** Callback can re-add itself for periodic execution.

**Internal Clock Operation:**

* Initial scheduling occurs at `step_period_ms` interval.
* Each callback processes sequencer state and reschedules itself.
* Handles adaptive delay to compensate for processing time.

**External Clock Supervision:**

* Runs periodically to detect MIDI clock timeout.
* Falls back to internal clock if no MIDI clock tick is received within the timeout threshold.

* Scheduling occurs in the async context.
* Worker callback `note_worker_enqueue_pending()` [note_scheduler.c L48](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/note_scheduler.c#L48-L48)  runs at the scheduled time.

* This two-stage design avoids USB mutex contention from async/IRQ context.

## Timing Precision and Constraints

### Resolution and Accuracy

| Metric | Value | Notes |
| --- | --- | --- |
| Time unit | Microseconds (µs) | Absolute timestamps use `time_us_64()` |
| Scheduling resolution | ~1µs | Limited by RP2350 timer hardware |
| Typical jitter | <100µs | Depends on callback processing time |

### Performance Considerations

**Callback Execution Context:**

* Workers may execute in IRQ context (CYW43 background IRQ).
* Must complete quickly to avoid blocking other callbacks.
* Long operations should defer to the main loop (e.g., using the note scheduler pattern).

**Sources:** [note_scheduler.c L1-L9](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/note_scheduler.c#L1-L9)

## Thread Safety

### CYW43 Configuration

The CYW43 async context provides built-in thread safety:

* Worker additions/removals are atomic from any thread.
* Callback execution is serialized through the CYW43 IRQ handler.
* Safe to call from Core 0 or Core 1.

### Threadsafe Background Configuration

The `async_context_threadsafe_background_t` implementation:

* Creates a dedicated background thread for worker processing.
* Uses mutex protection for worker queue operations.
* Safe for multi-core scheduling.

**Sources:** [async_timer.c L10-L33](https://github.com/Jus-Be/orinayo-pico/blob/6dde5a75/async_timer.c#L10-L33)
